# /plan-novel — 小說專案初始化

## 指令目標

將使用者的初步構想轉化為結構化的專案配置，在 `.novel-os/novels/[編號-故事名稱]/` 目錄下建立五個核心配置檔案，讓 AI 在後續撰寫時能遵循一致的故事核心與風格。

### 目錄命名規則
- 格式：`[三位數編號]-[kebab-case英文名]`
- 範例：`001-the-last-bookshop`、`002-chicago-noir`
- 編號：檢查 `.novel-os/novels/` 下現有目錄，取最大編號 +1；若目錄不存在，從 `001` 開始

---

## 執行前：資訊蒐集

若使用者的描述不夠完整，請先詢問以下資訊（一次問完，不要分多輪）：

1. **故事核心**：這部小說想說什麼？主角是誰？核心衝突是什麼？
2. **主題與意義**：這部作品對作者的意義是什麼？想傳達什麼？
3. **目標讀者**：誰會讀這本書？
4. **寫作風格**：
   - 敘事視角（第一人稱 / 第三人稱限知 / 第三人稱全知）
   - 時態（過去式 / 現在式）
   - 散文風格（精煉簡練 / 抒情詩意 / 對話驅動 / 其他）
5. **類型**：文學小說 / 推理 / 奇幻 / 愛情 / 科幻 / 其他

---

## 執行步驟

收到足夠資訊後，執行：

1. 決定編號（掃描 `.novel-os/novels/` 現有目錄）
2. 建立目錄：`.novel-os/novels/[編號-故事名稱]/`
3. 同步建立章節目錄：`.novel-os/novels/[編號-故事名稱]/chapters/`
4. 依序建立以下五個配置檔案：

---

### 1. `premise.md` — 故事核心

記錄完整的創作願景。格式如下：

```markdown
# [小說標題] — 故事核心

## 一句話前提
[用一句話說明故事：[主角]必須[做什麼]，否則[後果]。]

## 故事完整願景
[3-5段詳細描述故事的全貌]

## 核心主題
- [主題一]
- [主題二]
- [主題三]

## 這部作品的意義
[對作者個人而言，這部作品代表什麼]

## 目標讀者
[描述理想讀者]

## 類型與市場定位
[類型 + 相似作品參考]
```

### 2. `premise-lite.md` — AI 上下文摘要

`premise.md` 的精簡版，控制在 300 字以內，優先保留：
- 一句話前提
- 核心衝突
- 主要角色（名字 + 一行描述）
- 最重要的 2-3 個主題

> 用途：每次對話開始時載入，減少 Token 消耗，維持一致性。

### 3. `writing-style.md` — 小說專屬風格

```markdown
# [小說標題] — 寫作風格設定

## 敘事視角（POV）
[視角設定與規則]

## 時態
[過去式/現在式，及使用規則]

## 敘事聲音
[語氣、用詞習慣、句子長度傾向]

## 散文風格
[風格描述 + 2-3個示例句子]

## 對話處理
[對話標記方式、角色說話習慣]

## 禁止事項
[這部作品中應避免的寫作手法]
```

### 4. `writing-plan.md` — 寫作計畫

```markdown
# [小說標題] — 寫作計畫

## 專案概況
- 專案目錄：.novel-os/novels/[編號-故事名稱]/
- 預估字數：
- 預計章節數：
- 目標完成日期：

## 創作階段

### 階段一：大綱期
- [ ] 完成 story-outline.md
- [ ] 完成角色檔案
- [ ] 完成世界觀設定
- 目標日期：

### 階段二：初稿期
- [ ] 第一幕（chapter-01.md ～ chapter-XX.md）
- [ ] 第二幕（chapter-XX.md ～ chapter-XX.md）
- [ ] 第三幕（chapter-XX.md ～ chapter-XX.md）
- 目標日期：

### 階段三：修稿期
- [ ] 整體結構調整
- [ ] 風格一致性校對
- 目標日期：

## 每日/每週目標
[字數或場景數目標]
```

### 5. `decisions.md` — 創意決策紀錄

```markdown
# [小說標題] — 創意決策紀錄

## 使用說明
此檔案記錄創作過程中的重要決策，防止 AI 在後續建議中偏離軌道。

---

## [日期] — 專案初始化

**決策**：[初始核心設定的描述]
**理由**：[為什麼這樣決定]
**影響**：[這個決定如何影響後續創作]

---
```

---

## 執行完成後

1. 列出所有已建立的檔案路徑，清楚標示專案目錄名稱（含編號）
2. 簡短說明每個檔案的作用
3. 詢問使用者是否需要調整任何設定
4. 建議下一步：執行 `/create-outline` 來規劃詳細大綱

## 錯誤處理

- 若同名故事目錄已存在（相同 kebab-case 名稱）：告知使用者，詢問是否要覆蓋、改用新名稱，或進入現有專案繼續編輯
- 若資訊嚴重不足：最多詢問兩輪，之後用合理預設值填充並告知使用者哪些欄位待補充
