## 三層結構的具體運作方式如下：
1. **第一層：寫作標準 (Standards) — 定義「如何寫」**
    - 這一層是一次性設定、處處套用的基礎，建立了作者個人的風格框架。
    - 核心內容： 包含敘事聲音 (Narrative voice)、寫作視角 (POV)、散文風格、類型指南 (Genre Guides) 以及敘事技巧（如角色發展與節奏控制）。
    - 運作目標： 確保不論創作哪部作品，AI 產出的文字都能聽起來像作者本人的聲音。

2. **第二層：小說專案 (Novel) — 定義「寫什麼」**
    - 這一層專注於特定作品的創意願景。
    - 核心內容： 包含故事前提 (Premise)（作品的主旨與意義）、寫作計畫（階段性目標與完成期限）、創意決策紀錄以及該部小說專屬的風格設定。
    - 運作目標： 讓 AI 理解當前專案的具體核心，避免在創作過程中偏離作者的初始構想。

3. **第三層：稿件管理 (Manuscripts) — 定義「接下來寫什麼」**
    - 這是最細緻的寫作路線圖，將大構思轉化為可執行的任務。
    - 核心內容： 包含故事大綱（情節結構與角色弧線）、人物檔案（發展與關係）以及寫作任務（逐場戲的具體目標）。
    - 運作目標： 提供 AI 精確的執行藍圖，使其在撰寫具體場景時能維持角色一致性並推動劇情。

## 運作流程與協作方式
- 這三層結構的設計邏輯如同向人類寫作夥伴進行簡報，透過層層堆疊建立 AI 對創意過程的完整理解：
    - 層級關係： 系統依照「寫作標準 → 小說專案 → 稿件管理」的階層傳遞資訊。
    - 指令觸發： 使用者可透過指令（如 /plan-novel）初始化小說專案，接著用 /create-outline 生成詳細計畫，最後透過 /write-scenes 進行撰寫。在此過程中，AI 會同時參考這三層背景，自動追蹤角色細節並更新進度。

## 四大核心指令
Novel-OS 在整合進 Claude Code, Gemini CLI 或 Cursor 等 AI 工具後，主要提供四個核心指令。這些指令對應小說創作的不同階段，旨在將 AI 從單純的助手轉變為理解你寫作風格與故事結構的合作夥伴。
以下是這四個指令的詳細說明與範例：
1. **`/plan-novel` (小說專案初始化)**
	- 作用： 用於開啟一個全新的小說專案。它負責定義作品的「核心願景」，包含故事前提、寫作計畫、創意決策紀錄以及該作品專屬的風格設定。
	- 詳細內容： 
		1. **指令核心目標**: `/plan-novel` 的作用是將 AI 從一個「混亂的助手」轉變為理解你「專案全貌」的合作夥伴。它會根據你提供的初步構想，自動在專案根目錄下建立結構化的配置檔案，確保 AI 在後續撰寫時能遵循一致的故事核心與風格。
		2. **自動生成的五大核心檔案**: (於 .novel-os/novel/ 目錄)執行該指令後，系統會自動建立以下檔案，每個檔案都有其特定的上下文功能：
			- premise.md (故事核心)：記錄小說的完整願景、主旨、主題以及這部作品對作者的意義。
			- premise-lite.md (AI 上下文摘要)：這是 premise.md 的精簡版本，專門提供給 AI 作為高密度的背景資訊參考，旨在減少 Token 消耗並維持核心一致性。
			- writing-style.md (小說專屬風格)：定義這部小說專用的敘事聲音、人稱視角 (POV) 與語氣。這會覆蓋或補充第一層（Standards）的通用寫作標準，以符合特定故事的需求。
			- writing-plan.md (寫作計畫)：設定小說創作的各個階段（如大綱期、初稿期）、里程碑以及完成目標。
			- decisions.md (創意決策紀錄)：這是一個動態日誌，詳細記錄你在創作過程中做出的關鍵選擇（如劇情轉折、角色變動）及其理由，防止 AI 在後續建議中偏離軌道。
		3. **使用範例**: 當你要開始寫一部新作品時，可以輸入類似以下的詳細指令：
			- 指令： `/plan-novel` 我想寫一部關於一名女子繼承祖母在海邊小鎮書店的文學小說。核心主題：家族遺產、小鎮社群、第二次機會。目標讀者：喜歡角色驅動故事的成年讀者。寫作風格：第三人稱限知、抒情散文、現在時態。
		4. **運作邏輯與技術實現**:
			- 整合層級：該指令啟動後，AI 會讀取你在 /instructions/plan-novel.md 中的執行程序，並將產出的結果寫入本地檔案系統。
			- Skill 化建議：若將其轉換為標準 Agent Skill 格式，其 SKILL.md 的描述（Description）應包含關鍵字，以便 Claude 在你提到「我想開始寫新故事」時自動觸發該指令。
			- 優化輸出：在使用此指令前，建議先執行 /output-style novel-os-assistant，這會優化 Claude Code 跟 Gemini CLI 對小說創作工作流的響應模式。

2. **`/analyze-manuscript` (既有稿件導入與分析)**
	- 作用： 專為「將現有創作遷移至 Novel-OS 框架」而設計的指令。它能讓 AI 深度理解你已經寫好的內容，並自動完成結構化配置, 它屬於二層上下文結構中的第三層： 小說專案 (Novel)。
	- 詳細內容： AI 會掃描現有的章節與筆記，自動偵測你的寫作風格（語氣、視角、類型），並將已完成的工作整合到新的寫作計畫中。
		1. **指令核心作用**:此指令的主要目的是打破「新專案才適用」的限制，讓 AI 掃描並吸收你目前正在進行的小說專案（不論進度到哪裡），將其轉化為 Novel-OS 的三層上下文結構。
		2. **AI 會分析哪些具體內容?**: 當執行 `/analyze-manuscript` 時，AI 會針對以下四個面向進行深度掃描：
			- 目前的檔案結構 (Current Structure)：AI 會檢查你的檔案路徑、章節劃分方式以及目前的組織邏輯。
			- 寫作風格偵測 (Writing Style Detection)：
				- 敘事聲音 (Voice)：辨識你的用詞習慣與筆觸。
				- 敘事視角 (POV)：判斷是第一人稱、第三人稱限知或全知。
				- 類型鑑定 (Genre)：判斷作品屬於推理、奇幻、文學小說等哪種範疇。
			- 既有文件紀錄 (Documentation)：AI 會讀取你現有的角色筆記、設定集或大綱，確保系統紀錄能反映你已建立的創意內容。
			- 進度整合 (Integration)：AI 會將已完成的章節對應到「寫作計畫」中，讓你清楚知道目前的進度與剩餘的工作量。
		3. **執行後的產出結果**: 分析完成後，Novel-OS 會在你的專案目錄中自動建置以下結構：
			- 初始化 .novel-os/ 資料夾：建立專案專屬的隱藏配置空間。
			- 生成小說層 (Novel Layer) 檔案：包含 premise.md（故事核心）與 writing-style.md（偵測到的風格）。
			- 更新寫作計畫：在 writing-plan.md 中將已寫完的內容標記為完成，並設定後續里程碑。
			- 建立初步大綱：根據現有內容在 manuscripts/ 資料夾下建立初步的 story-outline.md。
		4. **使用範例與應用場景**:
			- 應用場景：你已經在 Word 或其他工具寫了三章小說，現在想換到 Claude Code 跟 Gemini CLI 使用 Novel-OS 來管理後續創作。
		5. **提示建議**: 為了獲得最佳效果，來源建議你在執行此指令前，確保你的寫作標準（Global Standards）已經初步設定完成。這樣 AI 在分析你的稿件時，能更精確地比對「你的理想風格」與「目前實際成品」之間的差異。

3. **`/create-outline` (故事規劃與大綱生成)**
	- 作用：  Novel-OS 系統中負責「故事規劃」的核心指令。它屬於三層上下文結構中的第三層：稿件管理 (Manuscripts)，旨在將宏觀的故事構思轉化為具體、可執行的寫作路線圖。
	- 詳細內容： 生成完整的故事大綱（包含情節結構與節奏）、角色檔案（弧線與關係）、世界觀設定，以及逐場戲的寫作任務清單。
		1. **指令核心作用**: 此指令的作用是為小說建立詳細的大綱與開發文件，確保 AI 在撰寫具體場景前，已經完整掌握了故事的結構 (Structure)、人物 (Characters) 與世界觀 (World-building)。它能防止劇情偏離軌道，並為之後的 /write-scenes 提供精確的執行藍圖。
		2. **自動產出的文件結構**: 執行 `/create-outline` 後，系統會在專案目錄 .novel-os/manuscripts/[日期-故事名稱]/ 下生成以下檔案：
			- story-outline.md (完整故事大綱)：包含詳盡的情節結構、故事節奏（Pacing）以及各階段的轉折點。
			- story-outline-lite.md (AI 上下文摘要)：大綱的精簡版，專門提供給 AI 作為高密度的背景資訊，以節省 Token 消耗並維持創作一致性。
			- sub-specs/character-profiles.md (角色檔案)：詳細紀錄角色的發展弧線 (Arcs)、人物關係以及性格成長細節。
			- sub-specs/world-building.md (世界觀設定)：包含故事背景的細節描寫、環境氛圍以及特定場景的設定。
			- tasks.md (寫作任務清單)：將整部小說拆解成逐場戲 (Scene-by-scene) 的具體寫作目標。這是 AI 撰寫場景時最直接的依據。
		3. **使用範例**: 當你已經有了基本的小說計畫，想進一步細化內容時：
			- 指令： /create-outline 為我的 1920 年代芝加哥推理小說建立詳細大綱。重點放在偵探調查私釀酒館謀殺案的過程，並加入與黑幫頭目對峙的情節。
		4. **最佳實踐建議**:
			- 仔細審查大綱：來源指出規劃階段決定了成品的品質，因此在進入撰寫階段前應反覆確認 story-outline.md 是否符合預期。
			- 角色驅動優先：建議先確保角色檔案 (Character Profiles) 的完整性，因為強大的角色設定是驅動精彩情節的關鍵。
			- 處理情節漏洞：如果在撰寫過程中發現邏輯問題，應回頭修正 story-outline.md 或 tasks.md，而非直接在場景中硬寫。

4. **`/write-scenes` (場景撰寫與進度管理)**
	- 作用： Novel-OS 系統中負責「執行撰寫任務」的核心指令。它屬於三層上下文結構中的第三層：稿件管理 (Manuscripts)，是將所有前期的規劃（風格、大綱、角色設定）轉化為實際小說正文的關鍵步驟。
	- 詳細內容： 根據前述的「寫作標準」與「故事大綱」來撰寫場景。它能確保角色的一致性（對話與行為），並在撰寫完成後自動更新里程碑與進度。
		1. **指令核心作用**: `/write-scenes` 的主要任務是根據前兩層（寫作標準與小說計畫）的設定，精確執行在第三層（稿件管理）中定義的逐場戲寫作任務 (Scene-by-scene tasks)。它確保 AI 在產出文字時，不僅僅是生成隨機的故事，而是像作者本人一樣在既定的框架內創作。
		2. **五大運作特色**: 當執行此指令時，系統會同時調用多個上下文來源來達成以下效果：
			- 精確風格匹配 (Exact Style Matching)：AI 會嚴格遵循他在第一層（Standards）學到的敘事聲音、語氣與技巧。
			- 角色一致性追蹤 (Character Consistency)：自動參考角色檔案（Character Profiles），確保角色的對話習慣、行為模式與成長弧線在不同場景中保持統一。
			- 劇情推進導向 (Plot Advancement)：根據 story-outline.md 的結構撰寫，確保每一場戲都能有效推動情節，不偏離主線。
			- 自動進度追蹤 (Progress Tracking)：撰寫完成後，AI 會自動更新寫作計畫（Writing Plan）中的里程碑，並在 tasks.md 中標記該任務已完成。
			- 會話管理 (Session Management)：具備清晰的開始與停止點，方便作者分多次完成長篇創作。
		3. **常見應用場景與範例**:指令非常靈活，可應用於不同的撰寫需求：
			- 撰寫具體場景：
			- 接續先前進度：
			- 指定章節目標：
		4. **相關檔案結構**: 為了支撐 `/write-scenes` 的運作，系統會參考並操作以下路徑的檔案：
			- 指令定義：~/.novel-os/instructions/write-scenes.md（負責協調寫作流程）與 write-scene.md（負責單一場景的具體撰寫規則）。
			- 任務來源：.novel-os/manuscripts/[日期-故事名稱]/tasks.md。
			- 產出目標：通常會將內容寫入專案根目錄下的 chapters/ 資料夾（如 chapter-01.md）。
		5. **最佳實踐建議**:
			- 每次專注於單一場景：為了維持最高品質，建議每次指令只要求 AI 完成一個具體的場景任務，而非整章。
			- 定期審閱與精煉：在每一場戲完成後進行審查，利用 Novel-OS 的風格審查功能來確保文字質感。
			- 信任結構：當你發現 AI 寫得不順時，通常是 tasks.md 或 story-outline.md 定義不夠清楚，應回頭修正大綱而非直接修改正文。
